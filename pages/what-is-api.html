What is an API?

Before diving into FHIR, you need to understand APIs — because FHIR is built on them. The good news: APIs are much simpler than they sound.

🍽️ Start with an Analogy
Think of an API like a waiter
🍽️
The Restaurant Analogy
Imagine you're at a restaurant. You (the customer) can't walk into the kitchen to get your food — that would be chaotic. Instead, you tell the waiter what you want. The waiter goes to the kitchen, gets your food, and brings it back to you.

In software: You = your app, Waiter = API, Kitchen = another system's data.
The API takes your request, goes to the data source, and returns the information you asked for.
📱

The App (You)

The thing making the request. Could be a hospital portal, an EHR system, a mobile app, or a data analytics tool — anything that needs information.

🔌

The API (The Waiter)

The "messenger" that carries requests and responses between systems. It defines the rules: what you can ask for, how to ask it, and what format you'll get back.

🗄️

The System (The Kitchen)

The system that holds the actual data — like an EHR (Epic, Cerner), a lab system, or a pharmacy system. It processes your request and sends the data back.

🏥 In Healthcare
A real-world API example: Getting a patient's lab results
1

A doctor opens the patient portal

The portal app needs to show the patient's recent blood test results. It sends a request to the EHR's API: "Give me the lab results for patient ID 12345."

2

The API receives the request

The API checks: Is this request properly formatted? Does the requester have permission? Is the patient ID valid? If everything is okay, it passes the request to the EHR database.

3

The EHR retrieves the data

The EHR looks up patient 12345 in its database and finds all the lab results. It packages this data in a structured format (JSON in FHIR's case).

4

The API returns the data

The API sends the lab results back to the portal. The portal reads the data and displays it in a readable way for the doctor. All of this happens in under a second.

ℹ️
Key Insight for BAs
When you write requirements like "the system shall display the patient's most recent lab results," you are implicitly requiring an API call. Understanding APIs helps you ask the right questions: What data is available? How often can we call the API? What happens if it fails?
📚 Key Concepts
Terms you'll hear and what they mean
GET

GET Request — "Give me data"

The most common type. Used to retrieve information. Think of it as asking: "Can I see the patient's record?" Nothing changes — you're just reading.

💡
Example: GET /Patient/12345 → Returns the patient record for ID 12345
POST

POST Request — "Create new data"

Used to create something new. Think of it as submitting a new patient registration form — new data gets added to the system.

💡
Example: POST /Patient → Creates a brand-new patient record
PUT

PUT Request — "Update existing data"

Used to update an existing record. Like changing a patient's address or phone number in the system — you're replacing old information with new.

💡
Example: PUT /Patient/12345 → Updates the full record for patient 12345
DELETE

DELETE Request — "Remove data"

Used to delete a record. In healthcare, this is rare and tightly controlled due to legal requirements for data retention.

⚠️
Healthcare systems rarely allow true deletion due to HIPAA and audit requirements.
🔗 Important Term
What is an "endpoint"?

You'll hear this word constantly in API conversations. It simply means the specific address (URL) you send your request to.

📮
Think of it like a mailing address
Just like you send a letter to "123 Main St, Apt 4B", you send an API request to a specific URL (endpoint). Each endpoint does a specific thing. The FHIR standard defines exactly what these addresses look like.

Example FHIR endpoints:
• GET /Patient/123 → Get patient with ID 123
• GET /Observation?patient=123 → Get all observations for patient 123
• POST /Appointment → Schedule a new appointment
