FHIR Basics

What is HL7 FHIR, why was it created, and why does it matter for you as a healthcare business analyst?

🔤 Breaking it Down
What does "HL7 FHIR" actually stand for?
🏛️

HL7

Health Level Seven International — a non-profit organization founded in 1987 that creates standards for sharing healthcare data. Think of them as the "rules committee" for healthcare IT.

The "Seven" refers to the 7th layer (Application Layer) of the ISO network model — the layer where healthcare applications operate.

⚡

FHIR

Fast Healthcare Interoperability Resources — HL7's modern standard for exchanging healthcare information. Released in 2014, it's now the required standard for US healthcare APIs.

  • ⚡ Fast — Quick to implement compared to older standards
  • 🏥 Healthcare — Designed specifically for medical data
  • 🔄 Interoperability — Systems can work together
  • 📦 Resources — Data is organized into "resources" (like Patient, Lab Result)
⚖️

R4 (the version)

FHIR has versions: DSTU2, STU3, R4, and R4B. R4 is the current stable standard used by most US EHRs (Epic, Cerner, etc.) and required by CMS regulations since 2021.

✅
When someone says "FHIR" in a project context today, assume they mean FHIR R4 unless specified otherwise.
🕰️ The Problem Before FHIR
Healthcare data was stuck in silos

To understand why FHIR matters, you have to understand the chaos it replaced.

⚠️
Before FHIR: The Old Reality
• Hospital A uses Epic. Clinic B uses Cerner. Neither could easily share patient data.
• A patient's records from 5 different providers were in 5 different, incompatible systems.
• Every new integration required expensive, custom-built connections between systems.
• Patients had to fill out the same forms repeatedly because records couldn't travel with them.

❌ Before FHIR

!

HL7 v2 — Inconsistent

The old standard allowed too many variations. Two hospitals using "HL7 v2" still needed custom code to talk to each other.

!

CCD/CDA — Hard to use

The XML-based format that came before FHIR was clunky, verbose, and difficult for modern apps to consume.

!

Expensive integrations

Hospital IT teams spent millions building point-to-point connections. Every new system meant a new custom integration project.

✅ After FHIR

1

One universal format

Patient data looks the same no matter which EHR it comes from. A Patient is always a Patient, with the same fields in the same structure.

2

Modern web technology

FHIR uses REST APIs and JSON — the same tech that powers Google, Amazon, and every modern web app. Developers already know how to use it.

3

Mandated by law (US)

The 21st Century Cures Act requires all EHRs to expose FHIR APIs. This means the data is now available by law, not just by luck.

🏗️ How FHIR is Built
Three things that make FHIR work
📦

1. Resources

FHIR breaks healthcare data into "Resources" — like building blocks. Each resource covers one concept: Patient, Appointment, Medication, Lab Result (Observation), Diagnosis (Condition), etc.

~150 resource types exist, covering nearly every aspect of healthcare.

🌐

2. REST API

FHIR uses a RESTful API model — meaning you use simple HTTP requests (GET, POST, PUT, DELETE) with standard web URLs to access data. No proprietary protocols needed.

GET https://ehr.hospital.com/fhir/Patient/123
📄

3. JSON Format

FHIR data is represented in JSON — a lightweight, human-readable format. We'll do a full JSON deep-dive in Module 4, but it's essentially structured like a well-organized list of key-value pairs.

"gender": "female"
🌍 Real World
Where you'll see FHIR in your projects

🏥 EHR Integrations

Epic, Cerner, Oracle Health, and all major EHRs now expose FHIR APIs. When your project needs to pull patient data from an EHR, FHIR is the API standard you'll use.

📱 Patient-Facing Apps

Apple Health, CommonHealth, and patient portal apps use FHIR's SMART on FHIR standard to let patients access their own records from any provider.

🔬 Care Gap & Analytics

Population health and care gap analysis tools pull FHIR data from EHRs to identify patients who need preventive care, follow-ups, or chronic disease management.

💊 Medication Management

Pharmacy systems, prescription monitoring programs, and medication reconciliation tools use FHIR's Medication and MedicationRequest resources.
